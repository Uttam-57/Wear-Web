Backend API service built with Node.js, Express, and MongoDB.

## Scope

This file covers backend setup and implementation details only.

- Product usage and role flow: [../README.md](../README.md)

- Node.js (ESM)
- Express 5
- Mongoose
- Zod
- JWT utilities
- Helmet, CORS, express-rate-limit, xss sanitization
- Stripe
- Cloudinary
- Nodemailer, Twilio
- Winston + Morgan

## Architecture

### Boot Sequence

Boot entry: src/server.js

1. DNS resolver setup
2. MongoDB connection
3. Express server listen
4. Scheduler registration

### App Composition

App entry: src/app.js

Middleware order requirements:

- payment webhook raw parser before express.json
- security middleware before route handlers
- sanitize middleware before controllers
- centralized error handler last

### Module Structure

Each module follows:

- moduleName.routes.js
- moduleName.validation.js
- moduleName.controller.js
- moduleName.service.js
- moduleName.model.js

### Error Handling

Central handler normalizes:

- mongoose duplicate key, cast, validation
- jwt expiration
- zod validation
- custom operational errors

### Scheduler

Registered scheduled jobs:

- wallet reconciliation job
- analytics snapshot job

Schedulers are not started in test environment.

## Backend Request Lifecycle

1. Request enters Express app.
2. Route-level parser selection is applied (raw for webhook, json/urlencoded for standard APIs).
3. Security middleware executes (helmet/cors/rate-limit/sanitize path).
4. Auth middleware resolves user identity and role permissions when required.
5. Validation middleware enforces request schema constraints.
6. Controller delegates business logic to service layer.
7. Service performs data access and cross-module side effects.
8. Response formatter returns payload or forwards error to centralized handler.

## Authentication and Authorization Model

- JWT access/refresh strategy is used for session continuity
- auth middleware attaches authenticated user context to request
- role checks gate customer, seller, and admin route groups
- hold/block/deletion states are evaluated in protected operations

## Module Interaction Patterns

### Order, Payment, and Inventory Coupling

- order placement reserves/decrements inventory
- payment confirmation finalizes paid state
- payment failure/cancellation path can restore inventory
- delivery confirmation can trigger wallet-credit reconciliation

### Return and Refund Coupling

- return transitions are state-guarded
- pickup confirmation uses OTP checks
- refund completion updates financial ledgers and return closure state

### Wallet and Payout Coupling

- payout requests are seller-initiated and admin-resolved
- approval debits/settles from eligible balance
- rejection records reason and preserves balance

## Data and Validation Strategy

- input contracts are validated using Zod schemas
- business invariants are enforced in services, not controllers
- mongoose models keep persistent shape and index-level constraints
- duplicate key and cast errors are normalized by global error handler

## Webhook Handling Requirements

- Stripe webhook route must read raw body for signature verification
- do not move webhook route behind generic JSON parser
- invalid signature events must return safe failure responses

## Operational Logging and Diagnostics

- request logs are generated by Morgan and forwarded to Winston
- structured error logs are persisted for incident triage
- use log correlation by request timestamp + route + actor id where available
- avoid logging secrets, tokens, or card-related payload fragments

## Resilience and Safety Controls

- rate-limit strategy differs for auth-sensitive and high-volume endpoints
- sanitize middleware protects against common payload injection patterns
- centralized AppError pattern keeps operational errors explicit
- test mode bypasses selective controls only where intentionally configured

| --- | --- | --- |
| NODE_ENV | Yes | Runtime mode |

| MONGO_URI_APP | Yes | MongoDB URI |

## Technical Notes

### Security and Input Processing

- sanitize middleware processes body/query/params
- webhook route bypasses JSON sanitize path where raw payload is required
- rate limiter policies differ by endpoint type

### Logging

- request logs via Morgan -> Winston
- application logs in:
  - logs/error.log
  - logs/combined.log

### Test Environment Behavior

- scheduler jobs are disabled
- some integrations use test-safe branches
- rate limiter is bypassed for test runtime

### OTP Behavior Notes

- OTP is used in delivery and return pickup confirmation paths
- expiry, retry limits, and lock windows are enforced by service rules
- in test mode, OTP may be included in responses for automation
- in non-test modes, OTP value exposure is suppressed

## Production Operations Notes

### Before Enabling Traffic

- run seed scripts only when environment bootstrapping is required
- verify webhook endpoint registration against active deployment URL
- verify scheduler intervals are safe for production load profile

### During Incident Triage

- inspect logs/error.log first for stack and route context
- verify DB connectivity and token secret consistency
- verify Stripe and Cloudinary credentials are not rotated unexpectedly
- compare failing path against middleware order in src/app.js

## Deployment Checklist

- Set all required environment variables.
- Configure CLIENT_URL to deployed frontend origin.
- Configure Stripe webhook secret and endpoint.
- Configure Cloudinary credentials.
- Verify MongoDB network and credentials.
- Run test suite before release.

## Troubleshooting

### API unreachable or failing

- Check MONGO_URI_APP and DB connectivity.
- Check CLIENT_URL and CORS mismatch.
- Check JWT secrets.

### Stripe webhook failures

- Check STRIPE_WEBHOOK_SECRET.
- Check webhook route receives raw body before JSON parsing.

### Upload failures

- Check Cloudinary credentials.
- Check upload middleware limits.

### Authorization failures

- Check Authorization header format: Bearer tokenValue.
- Check user role/status for protected routes.
